要約

Progress Events 仕様は、進捗状況（ progress ）の計測に利用できるイベント インタフェースを定める。 例えば HTTP entity body 転送など。 この仕様の主目的は、他の仕様からの利用にある。 The Progress Events specification defines an event interface that can be used for measuring progress; e.g. HTTP entity body transfers. This specification is primarily meant to be used by other specifications.

1. 序論

この節は参考である。 This section is non-normative.

この仕様は、進捗の計測に利用できるイベント インタフェース — ProgressEvent — を定める。 他の仕様はその目的のためにこの仕様を利用する。 This specification defines an event interface — ProgressEvent — that can be used for measuring progress. Other specifications use this specification for that purpose.

この例ではリソース取得の経過を示すため、この仕様で定義される概念と組み合わせられた XMLHttpRequest と HTML progress 要素が併用される。 [XHR] [HTML] In this example XMLHttpRequest, combined with concepts defined in this specification, and the HTML progress element are used together to display the process of fetching a resource. [XHR] [HTML] 

<!DOCTYPE html>
<title>Waiting for Magical Unicorns</title>
<progress id="p"></progress>
<script>
  var progressBar = document.getElementById("p"),
      client = new XMLHttpRequest();
  client.open("GET", "magical-unicorns");
  client.onprogress = function(pe) {
    if(pe.lengthComputable) {
      progressBar.max = pe.total;
      progressBar.value = pe.loaded;
    }
  }
  client.onloadend = function(pe) {
    progressBar.value = pe.loaded;
  }
  client.send();
</script>

もちろん、完全に動作するコードでは、ネットワークエラーや利用者による中断など, より念入りかつ多様な状況に対応するものになるだろう。 Fully working code would of course be more elaborate and deal with more scenarios, such as network errors or the end user terminating the request.

3. 用語

この仕様の用語は DOM4 と HTTP に規定される。 [DOM] [HTTP] Terminology used in this specification is from DOM4 and HTTP. [DOM] [HTTP]

4. インタフェース ProgressEvent

[Constructor(DOMString type, optional ProgressEventInit eventInitDict)]
interface ProgressEvent : Event {
  readonly attribute boolean lengthComputable;
  readonly attribute unsigned long long loaded;
  readonly attribute unsigned long long total;
};

dictionary ProgressEventInit : EventInit {
  boolean lengthComputable;
  unsigned long long loaded;
  unsigned long long total;
};

ProgressEvent インタフェースを利用する イベント は、何らかの進捗を指示する。 Events using the ProgressEvent interface indicate some kind of progression.

lengthComputable 属性は初期化時の値を返さなければならない。 イベント 作成時においては、この属性は false に初期化されなければならない。 The lengthComputable attribute must return the value it was initialized to. When an event is created the attribute must be initialized to false.

loaded, total 属性は初期化時の値を返さなければならない。 イベント 作成時においては、いずれの属性も 0 に初期化されなければならない。 The loaded and total attributes must return the value they were initialized to. When an event is created the attributes must be initialized to 0.

4.1. ProgressEvent インタフェースを用いて HTTP のイベントを発火する

e 進捗イベントを発火する とは、次の条件を満たすように, ProgressEvent インタフェースを利用する イベント により 名前 e のイベントを発火する ことを意味する： To fire a progress event named e means to fire an event named e with an event using the ProgressEvent interface that also meets these conditions:

  • loaded 属性を転送された HTTP entity body バイト数に初期化する。 Initialize the loaded attribute to the number of HTTP entity body bytes transferred.

  • HTTP entity body の長さが Content-Length ヘッダを通して得られた場合、 lengthComputable 属性を true に, total 属性をその長さに初期化する。 If the length of the HTTP entity body is known through the Content-Length header, initialize the lengthComputable attribute to true and initialize the total attribute to the length.

注記： この定義は他の仕様にも用いられ得る。 例えば XMLHttpRequest など。 This definition can be used by other specifications. XMLHttpRequest does this for instance. [XHR]

4.2. 他の文脈下における ProgressEvent インタフェースの利用によるイベントの発火

これについては、その種の文脈を導入する仕様の編集者に委ねられる。 編集者には、この仕様, および ProgressEvent インタフェースを利用するイベントを組み込む他の仕様との整合性が保たれるように仕様を定めることが奨励される。 This is left as an exercise for the editor of the specification that introduces such a context. The editor is encouraged to define it in a way consistent with this and other specifications that utilize events using the ProgressEvent interface.

4.3. ProgressEvent インタフェースを利用するイベントに提案されている名前

この節は参考である。 This section is non-normative.

ProgressEvent インタフェースを利用する イベント に勧められる type 属性値を下の一覧に要約する。 仕様の編集者が、その固有の目的に応じて詳細を調整することは自由であるが、 public-webapps@w3.org にて、その利用について W3C WebApps Working Group と相談し, その問題に慣れた人達からの意見も念頭に入れることが強く奨励される。 The suggested type attribute values for use with events using the ProgressEvent interface are summarized in the table below. Specification editors are free to tune the details to their specific scenarios, though are strongly encouraged to discuss their usage with the W3C WebApps Working Group on public-webapps@w3.org to ensure input from people familiar with the subject.

type 属性値 説明 回数 時機
loadstart 進捗が開始された 1 最初。
progress 進捗中 0 以上

loadstart が 配送 された後。 After loadstart has been dispatched.

error 進捗に失敗した 0 または 1

最後の progress が 配送 された後か, または progress がまだ 配送 されていない場合は loadstart が 配送 された後。 After the last progress has been dispatched, or after loadstart has been dispatched if progress has not been dispatched.

abort 進捗が中止された 0 または 1
load 進捗が成功裡に終了 0 または 1
loadend 進捗が停止した 1

error, abort, load のいずれかが 配送 された後。 After one of error, abort, or load has been dispatched.

error, abort, load イベント型は互いに排他的である。 The error, abort, and load event types are mutually exclusive.

ウェブプラットフォーム全体に渡り、 error, abort, load イベント型においては bubbles, cancelable 属性は false に初期化される。 したがって, 整合性のために、 ProgressEvent インタフェースを利用するすべての イベント にも同じふるまいが勧められる。 Throughout the web platform the error, abort, and load event types have their bubbles and cancelable attributes initialized to false, so it is suggested that for consistency all events using the ProgressEvent interface do the same.

4.4. 保安上の考慮点

cross-origin リクエストにおいては、例えば Cross-Origin Resource Sharing のような何らかの類いのオプトインが、 ProgressEvent インタフェースを利用する イベント が 配送される 前に、あらかじめ用いられる必要がある。 さもなければ、取得できないような情報（例えばサイズ）も露呈されるので。 [CORS] For cross-origin requests some kind of opt-in, e.g. Cross-Origin Resource Sharing, has to be used before events using the ProgressEvent interface are dispatched as information (e.g. size) would be revealed that cannot be obtained otherwise. [CORS]
