1. 概要

各種 CSS プロパティの定義表における値定義の欄には、 キーワード, データ型（山括弧 <, > で括られて記される）, および それらの組み合せ方についての情報が含まれている。 この仕様では、多くのプロパティから利用し得る，汎用データ型（最も広範に利用されている <length> など）について述べる。 より固有のデータ型（例えば <spacing-limit> ）は、対応しているモジュールにて述べられる。 The value definition field of each CSS property can contain keywords, data types (which appear between < and >), and information on how they can be combined. Generic data types (<length> being the most widely used) that can be used by many properties are described in this specification, while more specific data types (e.g., <spacing-limit>) are described in the corresponding modules.

1.1. モジュール間の相互作用

このモジュールは [CSS21] の 1.4.2.1, 4.3, A.2 節にて定義されるデータ型を置き換え, 拡張する。 This module replaces and extends the data type definitions in [CSS21] sections 1.4.2.1, 4.3, and A.2.

2. 値定義の構文

ここに述べられる構文は、 CSS プロパティの妥当な値の集合を定義するために利用される。 プロパティ値は、1 個以上の成分を持ち得る。 【 この構文は、主に，プロパティ定義表の “値” の欄にて利用される。 】 The syntax described here is used to define the set of valid values for CSS properties. A property value can have one or more components.

2.1. 成分値の型

プロパティの成分値の型は、その値定義において，以下の方法で指定が与えられる： Component value types are designated in several ways:

  1. 引用符や山括弧を伴わずに, リテラルとして現れるものは、キーワード値である（ auto, disc, 等々）。 keyword values (such as auto, disc, etc.), which appear literally, without quotes (e.g. auto)
  2. 基本データ型は、山括弧（ <, > ）で括られて現れる（ <length>, <percentage>, 等々）。 basic data types, which appear between < and > (e.g., <length>, <percentage>, etc.).
  3. 同じ名前を持つプロパティと同じ範囲の値をとる，型（例えば <‘border-width’>, <‘background-attachment’> など）。 この場合、その型の名前は，山括弧の間の（引用符で完結する）プロパティ名になる。 ただし， inherit のような CSS 全域キーワード は、 【 どのプロパティもその種の値をとり得るとされているので， 】 この種の型の値域からは 除外される。 types that have the same range of values as a property bearing the same name (e.g., <‘border-width’>, <‘background-attachment’>, etc.). In this case, the type name is the property name (complete with quotes) between the brackets. Such a type does not include CSS-wide keywords such as inherit.
  4. 非終端記号は プロパティと同じ名前を共有しない。 この場合、非終端記号の名前は <spacing-limit> のように、山括弧（ <, > ）で括られて現れる。 <border-width> と <‘border-width’> との間の差異に注意： 後者は， border-width プロパティがとり得る値として定義される一方、前者は，他のどこかで 明示的に拡張が定義される必要がある。 非終端記号の定義は、概して，仕様の中でそれが最初に現れる近辺に示される。 non-terminals that do not share the same name as a property. In this case, the non-terminal name appears between < and >, as in <spacing-limit>. Notice the distinction between <border-width> and <‘border-width’>: the latter is defined as the value of the border-width property, the former requires an explicit expansion elsewhere. The definition of a non-terminal is typically located near its first appearance in the specification.

一部のプロパティの値定義は、［ スラッシュ（ / ）／カンマ（ , ）／丸括弧（ (, ) ） ］もリテラルとして含むことがある。 これらは、対応するトークンを表現する。 正符号（ + ）など，他の成分値に現れ得る非キーワードのリテラル文字は、一重引用符で括って記されなければならない。 Some property value definitions also include the slash (/), the comma (,), and/or parentheses as literals. These represent their corresponding tokens. Other non-keyword literal characters that may appear in a component value, such as “+”, must be written enclosed in single quotes.

文法の中でオプションの項を区切るために利用される カンマ は、一部の状況では，暗黙的に省略し得る。 また、文法において，［ プロパティや他の CSS 値の中のトップレベルのリスト, あるいは 関数の引数リスト ］の中に指定されたカンマは、次に挙げる場合には，省略されなければならない： Commas specified in the grammar are implicitly omissible in some circumstances, when used to separate optional terms in the grammar. Within a top-level list in a property or other CSS value, or a function’s argument list, a comma specified in the grammar must be omitted if:

  • カンマに先行するすべての項が省略された場合 all items preceding the comma have been omitted

  • カンマに後続するすべての項が省略された場合 all items following the comma have been omitted

  • カンマに挟まれた項が省略されると，複数のカンマが（空白やコメントを無視して）連続することになる場合 multiple commas would be adjacent (ignoring white space/comments), due to the items between the commas being omitted.

例えば，［ どれもオプションである， 3 個の引数 ］を受容する関数の文法は、次の様に記せる： For example, if a function can accept three arguments in order, but all of them are optional, the grammar can be written like: 

example( first? , second? , third? )

この文法の下では、 example(first, second, third) は妥当であり， example(first, second) ／ example(first, third) ／ example(second) も同様である。 一方で， example(first, , third) は、無効になる。 カンマには、 2 個のオプションを分離することが要求されるので。 同じ理由から、 example(,second) ／ example(first,) ／ example(first second) も無効になる。 Given this grammar, writing example(first, second, third) is valid, as is example(first, second) or example(first, third) or example(second). However, example(first, , third) is invalid, as one of those commas are no longer separating two options; similarly, example(,second) and example(first,) are invalid. example(first second) is also invalid, as commas are still required to actually separate the options.

カンマが暗黙的に省略し得ないものであったなら、省略し得る引数を正しく表記するための文法は，ずっと複雑になり，その特色機能の単純性も損なわれるであろう。 If commas were not implicitly omittable, the grammar would have to be much more complicated to properly express the ways that the arguments can be omitted, greatly obscuring the simplicity of the feature.

すべての CSS プロパティには、プロパティ値の単独の成分として， CSS 全域キーワード値が受容される。 可読性のため、これらのプロパティ値は，構文­定義には明示的に挙げられない。 例えば， border-color の値定義は（ <color>{1,4} と示されているが），実際には： [ <color>{1,4} ] | inherit | initial | unset である。 All CSS properties also accept the CSS-wide keyword values as the sole component of their property value. For readability these are not listed explicitly in the property value syntax definitions. For example, the full value definition of border-color is <color>{1,4} | inherit | initial | unset (even though it is listed as <color>{1,4}).

【 CSS 全域キーワードの節に付記されているように，新たな全域キーワードが導入されれば、それらも追加されることになるであろう（しかしながら，仮に導入された場合、 作者定義の識別子 との衝突から，後方互換性の問題も生じ得ることになる）。 】

注記： したがって、一般に，同じ宣言の中で これらのキーワードが他の成分値と組み合わされた場合、その宣言は無効になる。 例えば background: url(corner.png) no-repeat, inherit; は、無効である。 Note: This implies that, in general, combining these keywords with other component values in the same declaration results in an invalid declaration. For example, background: url(corner.png) no-repeat, inherit; is invalid.

2.2. 成分値の結合子

プロパティの値定義においては、各­成分値が次の組み合せで配列される: Component values can be arranged into property values as follows:

（結合子を挟まずに）並記された成分
それらの成分すべてが，示された順序で現れなければならないことを意味する。 Juxtaposing components means that all of them must occur, in the given order.
二重アンパサンド（ && ）
複数個の成分の区切り： それらの成分すべてが現れなければならないが，その順序は任意であることを意味する。 A double ampersand (&&) separates two or more components, all of which must occur, in any order.
二重縦線（ || ）
複数のオプションの区切り： それらのうち 1 個以上が現れなければならないが，その順序は任意であることを意味する。 A double bar (||) separates two or more options: one or more of them must occur, in any order.
縦線（ | ）
複数の選択肢の区切り： それらのうち，きっかり 1 個が現れなければならないことを意味する。 A bar (|) separates two or more alternatives: exactly one of them must occur.
角括弧（ […] ）
グループ化を表す。 Brackets ([ ]) are for grouping.

これらの結合順位は優先度の高いものから、［ 並記, 二重アンパサンド, 二重縦線, 縦線 ］の順になる。 従って、次の２行は等価になる： Juxtaposition is stronger than the double ampersand, the double ampersand is stronger than the double bar, and the double bar is stronger than the bar. Thus, the following lines are equivalent: 

  a b   |   c ||   d &&   e f
[ a b ] | [ c || [ d && [ e f ]]]

並び替え可能な結合子（ ||, && ）に対しては、文法における順序は重要でない — 同じグループ内の各­成分は、どのような順序で記されてもよい。 したがって，次の 2 行は等価になる： For reorderable combinators (||, &&), ordering of the grammar does not matter: components in the same grouping may be interleaved in any order. Thus, the following lines are equivalent: 

a || b || c
b || a || c

2.3. 成分値の量指定子

どの［ 値型 ／ キーワード ／ 角括弧で括られたグループ ］（以下、これらを “構文単位” と総称する）にも，次のいずれかの修飾子が後置され得る： Every type, keyword, or bracketed group may be followed by one of the following modifiers:

アスタリスク（ * ）
直前の構文単位の 0 回以上の出現を指示する。 An asterisk (*) indicates that the preceding type, word, or group occurs zero or more times.
正符号（ + ）
直前の構文単位の 1 回以上の出現を指示する。 A plus (+) indicates that the preceding type, word, or group occurs one or more times.
疑問符（ ? ）
直前の構文単位が任意選択であること（すなわち 0 回または 1 回の出現）を指示する。 A question mark (?) indicates that the preceding type, word, or group is optional (occurs zero or one times).
波括弧に括られた整数（ {A} ）
直前の構文単位の A 回の出現を指示する。 A single number in curly braces ({A}) indicates that the preceding type, word, or group occurs A times.
波括弧に括られたカンマ区切りの整数の組（ {A,B} ）
直前の構文単位の A 回から B 回までの出現を指示する。 B は省略されてもよく（ {A,} ），その場合は A 回 以上, 上限なしの出現を指示する。 A comma-separated pair of numbers in curly braces ({A,B}) indicates that the preceding type, word, or group occurs at least A and at most B times. The B may be omitted ({A,}) to indicate that there must be at least A repetitions, with no upper bound on the number of repetitions.
ハッシュ記号（ # ）
直前の構文単位がカンマ­トークンで区切られた上で 1 回以上 出現することを指示する（カンマの前後に空白やコメントが現れてもよい）。 これには任意選択で上述の波括弧 形が後続し得る（例えば <length>#{1,4} ）。 その場合、より詳細な出現回数を指示する。 A hash mark (#) indicates that the preceding type, word, or group occurs one or more times, separated by comma tokens (which may optionally be surrounded by white space and/or comments). It may optionally be followed by the curly brace forms, above, to indicate precisely how many times the repetition occurs, like <length>#{1,4}.
グループの直後の感嘆符（ ! ）
そのグループが必須であり， 1 個以上の値を生成しなければならないことを指示する。 すなわち、グループ内の各項は文法により個別的に省略し得るとしても，全体としては成分値をすべて省略してはならない。 An exclamation point (!) after a group indicates that the group is required and must produce at least one value; even if the grammar of the items within the group would otherwise allow the entire contents to be omitted, at least one component value must not be omitted.

（ *, +, # で指示される）成分値の繰り返しに対しては、 UA は、少なくとも 20 回以上の繰り返しをサポートしなければならない。 プロパティ値が，サポートする数を超える成分の繰り返しを含む場合、その宣言は，無効であったかのように 無視されなければならない。 For repeated component values (indicated by *, +, or #), UAs must support at least 20 repetitions of the component. If a property value contains more than the supported number of repetitions, the declaration must be ignored as if it were invalid.

2.4. 結合子と量指定子の共通パタン

特定の個数と順序による，複数の独立な成分値を組み合せるための、共通の用法がいくつかある。 特に、仕様の策定者が，［ 作者は、成分値のある集合から，［ 文法に指定される順序か，あるいは任意の順序 ］で，その中の［ ゼロ個以上／ 1 個以上／すべて ］を選択しなければならない ］ことを表記したいことは、よくある。 There are a small set of common ways to combine multiple independent component values in particular numbers and orders. In particular, it’s common to want to express that, from a set of component value, the author must select zero or more, one or more, or all of them, and in either the order specified in the grammar or in any order.

これらはすべて、結合子や量指定子による単純なパタンを利用して，容易に表記できる： All of these can be easily expressed using simple patterns of combinators and multipliers:

順序 指定どおり 任意
ゼロ個以上 A? B? C? A? || B? || C?
1 個以上 [ A? B? C? ]! A || B || C
すべて A B C A && B && C

順序が “任意” のものは，どれも結合子を用いて表記される一方、 “指定どおり” のものは，どれも並記になっていることに注意。 Note that all of the "any order" possibilities are expressed using combinators, while the "in order" possibilities are all variants on juxtaposition.

2.5. 成分値と空白

特に指定されない限り、空白やコメントが，［ 上の結合子や量指定子を用いて結合された成分の前後や, その各­成分の合間 ］に現れてもよい。 Unless otherwise specified, white space and/or comments may appear before, after, and/or between components combined using the above combinators and multipliers.

注記： 多くの場合、互いのトークンを判別するため，成分­間のスペースが実際に要求される。 例えば，値 1em2em は、数字 1 と無効な単位­識別子 em2em を伴う，単独の <dimension-token> トークンとして構文解析されることになる。 この場合、 2 個の長さ値［ 1em, 2em ］として構文解析されるためには，数字 2 の前にスペースが要求される。 Note: In many cases, spaces will in fact be required between components in order to distinguish them from each other. For example, the value 1em2em would be parsed as a single <dimension-token> token with the number 1 and the identifier em2em, which is an invalid unit. In this case, a space would be required before the 2 to get this parsed as the two lengths 1em and 2em.

3. テキストデータ型

識別子とは、 <ident-token> [CSS3SYN] に適合する文字の並びである。 識別子は，引用符で括ることはできない。 さもなければ、文字列として解釈されることになる。 An identifier is a sequence of characters conforming to the <ident-token> grammar. [CSS3SYN] Identifiers cannot be quoted; otherwise they would be interpreted as strings.

【 明示的にこの意味の識別子を指す（他の文脈の “識別子” と区別する）ときには、 “CSS 識別子” とも記される。 】

3.1. 定義済みキーワード

値定義の欄の中では、定義済みの意味を伴うキーワードは，リテラルとして現れる。 キーワードは CSS 識別子 であり、 ASCII 文字大小無視 の下で解釈される（すなわち、 a 〜 z と A 〜 Z は等価である）。 In the value definition fields, keywords with a pre-defined meaning appear literally. Keywords are CSS identifiers and are interpreted ASCII case-insensitively (i.e., [a-z] and [A-Z] are equivalent).

例えば， border-collapse プロパティの値定義は次で与えられる： For example, here is the value definition for the border-collapse property:

値 collapse | separate

その用例： And here is an example of its use: 

table { border-collapse: separate }

3.1.1. CSS 全域キーワード： initial, inherit, unset

成分値の型にて定義されるように，すべてのプロパティは CSS 全域キーワード を受容する。 これらは、すべての CSS プロパティに共通する値の算出法を，表現する。 As defined above, all properties accept the CSS-wide keywords, which represent value computations common to all CSS properties.

initial キーワードは、プロパティの初期値として指定されている値を表現する。 inherit キーワードは、要素の親上のプロパティの算出値を表現する。 unset キーワードは、プロパティが継承するかどうかに応じて， inherit か initial としてふるまう。 これらのキーワードは Cascade モジュール [CSS3CASCADE] にて規定されている。 The initial keyword represents the value specified as the property’s initial value. The inherit keyword represents the computed value of the property on the element’s parent. The unset keyword acts as either inherit or initial, depending on whether the property is inherited or not. All of these keywords are normatively defined in the Cascade module. [CSS3CASCADE]

他の CSS 仕様も追加の CSS 全域キーワードを定義し得る。 Other CSS specifications can define additional CSS-wide keywords.

3.2. 作者定義の識別子： <custom-ident> 型

一部のプロパティは、成分値として，作者定義による任意の識別子を受容する。 この汎用データ型は <custom-ident> で表され、そのプロパティの値定義による定義済みキーワードに解釈されないような，任意の妥当な CSS 識別子 を表現する。 その種の識別子の文字の大小は区別される（例えば example と EXAMPLE は、互いに無関係な，作者定義による異なる識別子である）。 Some properties accept arbitrary author-defined identifiers as a component value. This generic data type is denoted by <custom-ident>, and represents any valid CSS identifier that would not be misinterpreted as a pre-defined keyword in that property’s value definition. Such identifiers are fully case-sensitive, even in the ASCII range (e.g. example and EXAMPLE are two different, unrelated user-defined identifiers).

プロパティ値において，位置により多義的になるようなキーワードを構文解析するとき、 <custom-ident> 生成規則は，［ そのキーワードに該当し得るような，未だ充足されていない生成規則 ］が他に無い場合にのみ，そのキーワードに該当し得る。 When parsing positionally-ambiguous keywords in a property value, a <custom-ident> production can only claim the keyword if no other unfulfilled production can claim it.

例えば、簡略形による宣言 animation: ease-in ease-out; は、冗長形による宣言 animation-timing-function: ease-in; animation-name: ease-out; に等価になる。 ease-in は、 animation-timing-function に属する <single-timing-function> 生成規則に該当する結果、 ease-out は， animation-name に属する <custom-ident> 生成規則に該当するようになる。 For example, the shorthand declaration animation: ease-in ease-out is equivalent to the longhand declarations animation-timing-function: ease-in; animation-name: ease-out;. ease-in is claimed by the <single-timing-function> production belonging to animation-timing-function, leaving ease-out to be claimed by the <custom-ident> production belonging to animation-name.

注記： <custom-ident> が伴われる文法を設計するときは、 プロパティ内のどのキーワード値とも競合し得なくなるように， <custom-ident> は，常に “位置により多義的にならない” ようにするべきである。 Note: When designing grammars with <custom-ident>, the <custom-ident> should always be "positionally unambiguous", so that it’s impossible to conflict with any keyword values in the property.

4. 数値データ型

プロパティは、数値を一定範囲に制約し得る。 許容­範囲外の値を伴う宣言は無効であり，無視されなければならない。 Properties may restrict numeric values to some range. If the value is outside the allowed range, the declaration is invalid and must be ignored.

8.1.1. 構文

calc() 関数の構文は： The syntax of a calc() function is: 

<calc()> = calc( <calc-sum> )
<calc-sum> = <calc-product> [ [ '+' | '-' ] <calc-product> ]*
<calc-product> = <calc-value> [ '*' <calc-value> | '/' <number> ]*
<calc-value> = <number> | <dimension> | <percentage> | ( <calc-sum> )

加えて，演算子 +, - の両側には空白が要求される（演算子 *, / の両側には要求されない）。 In addition, white space is required on both sides of the + and - operators. (The * and / operaters can be used without white space around them.)

UA は、 calc() 式の項数として，少なくとも 20 個まではサポートしなければならない。 ここで， NUMBER, DIMENSION, PERCENTAGE 【 <number>, <dimension>, <percentage> 】 のそれぞれを 1 項として数える。 項数がサポートされる数よりも多い場合、その式は，無効であったかのように扱われなければならない。 UAs must support calc() expressions of at least 20 terms, where each NUMBER, DIMENSION, or PERCENTAGE is a term. If a calc() expression contains more than the supported number of terms, it must be treated as if it were invalid.

8.1.2. 型検査

どの数式も、次のいずれかをその 解決型 として持つ： <length>, <frequency>, <angle>, <time>, <number>, <integer> 。 解決型は、式が置かれている場所で妥当な型でなければならない。 他の場合、その式は無効である。 式の解決型は、式に含まれている値の型により決定される。 <number-token> は、 <number> または <integer> 型になる。 <dimension-token> の型は、その単位により与えられる（ cm は <length> になり, deg は <angle> になる等々）。 attr() 式の型は、その <type-or-unit> 引数の型で与えられる。 A math expression has a resolved type, which is one of <length>, <frequency>, <angle>, <time>, <number>, or <integer>. The resolved type must be valid for where the expression is placed; otherwise, the expression is invalid. The resolved type of the expression is determined by the types of the values it contains. <number-token>s are of type <number> or <integer>. A <dimension-token>’s type is given by its unit (cm is <length>, deg is <angle>, etc.). An attr() expression’s type is given by its <type-or-unit> argument.

百分率が受容される文脈に置かれた式 の中の <percentage-token> は，その百分率が相対基準にしている値の型を持つ。 他の文脈­下における，百分率が含まれている数式は、無効である。 If percentages are accepted in the context in which the expression is placed, a <percentage-token> has the type of the value that percentages are relative to; otherwise, a math expression containing percentages is invalid.

演算子をとる下位式からは、その引数に基づいて型が得られる。 式を単純化するため、演算子が受容し得る引数­型には制約がある。 それぞれの演算子ごとに、左右両側の引数の型についての制約が検査される。 両者が互換であれば、型は以下に述べるように解決される（説明を簡潔にするため，演算子の優先順位規則については脇に置く）: Operators form sub-expressions, which gain types based on their arguments. To make expressions simpler, operators have restrictions on the types they accept. At each operator, the types of the left and right argument are checked for these restrictions. If compatible, the type resolves as described below (the following ignores precedence rules on the operators for simplicity):

  • +, - （加算, 減算）においては、［ 両側の引数が同じ型である ］か, または［ 一方が <number> で, 他方が <integer> である ］かどうかが検査される。 前者の場合は，その型に解決される。 後者の場合は， <number> に解決される。 At + or -, check that both sides have the same type, or that one side is a <number> and the other is an <integer>. If both sides are the same type, resolve to that type. If one side is a <number> and the other is an <integer>, resolve to <number>.
  • * （アスタリスク, 乗算）においては、両側の引数の少なくとも一方が <number> であるかどうかが検査される。 両者とも <integer> であれば， <integer> に解決される。 他の場合は，他方の型に解決される。 At *, check that at least one side is <number>. If both sides are <integer>, resolve to <integer>. Otherwise, resolve to the type of the other side.
  • / （スラッシュ, 除算）においては、右側の項が <number> であるかどうかが検査される。 左側の項が <integer> であれば <number> に解決される。 他の場合は，左側の項の型に解決される。 At /, check that the right side is <number>. If the left side is <integer>, resolve to <number>. Otherwise, resolve to the type of the left side.

上の検査に合格しない演算子を含む式は，無効である。 ゼロによる除算も無効になる。 これには、リテラルのゼロによる除算も, ゼロに評価される任意の数値式による除算も含まれる（純粋な数値式は、構文解析時に追加の情報なしで評価し得るので）。 If an operator does not pass the above checks, the expression is invalid. Also, division by zero is invalid. This includes both dividing by the literal number zero, as well as any numeric expression that evaluates to zero (as purely-numeric expressions can be evaluated without any additional information at parse time).

注記： calc() 式やその解決型の妥当性は、代数的な簡約により影響されることはない。 例えば， calc(5px - 5px + 10s) や calc(0 * 5px + 10s) は、いずれも長さと時間の加算を試みるので，不正である。 Note: Algebraic simplifications do not affect the validity of the calc() expression or its resolved type. For example, calc(5px - 5px + 10s) and calc(0 * 5px + 10s) are both invalid due to the attempt to add a length and a time.

8.1.3. 算出値

calc() 式の算出値は、すべての成分が算出された式になる。 The computed value of a calc() expression is the expression with all components computed.

算出値の時点で解決されない百分率は、 calc() 式の中でも解決されない。 例えば，式 calc(100% - 100% + 1em) は、 1em ではなく， calc(1em + 0%) に解決される。 値の中の百分率から，それを算出するための特別な規則（例えば height プロパティ）がある場合、その規則が calc() 式に含まれるどの百分率にも適用される。 Where percentages are not resolved at computed-value time, they are not resolved in calc() expressions, e.g. calc(100% - 100% + 1em) resolves to calc(1em + 0%), not to 1em. If there are special rules for computing percentages in a value (e.g. the height property), they apply whenever a calc() expression contains percentages.

注記： 直列化の規則は、算出の構造は保全しないので、実装は，値を内部的に格納するときには、式を，ここに要求される以上に単純化できる — 特に、すべての calc() 式は、単位が一致するものどうしの乗算／除算は消去して，［ <number>, <percentage>, いくつかの <dimension> ］の和に簡約できる。 Note: The serialization rules do not preserve the structure of the computation, so implementations can simplify the expressions further than what is required here when storing the values internally; in particular, all calc() expressions can be reduced to a sum of a <number>, a <percentage>, and some <dimension>s, eliminating all multiplication or division, and combining identical units.

この時点で、すべての単位は，算出値の時点で型ごとに単独の単位に絶対化できるので、 calc() 式は，［ <number>, <percentage>, ［ 適切な型の，単独の絶対 <dimension> ］］の和に簡約できる。 At this time, all units can be absolutized to a single unit per type at computed-value time, so at that point the calc() expression can be reduced to just a <number>, a <percentage>, and a single absolute <dimension> of the appropriate type.

例えば， background-position における百分率値は、長さ値とは異なる，特別なふるまいを持つ： For example, background-position has special behavior for percentage values, different from lengths: 

.foo {
	width: 200px;
	background-image: url(bg.png);
	background-position: left 50%;
	/* 200px の 50% が 100px であるにも関わらず */
	background-position: left 100px;
	/* これとは効果が異なる */
}

このため、 background-position における calc() の中の百分率は： background-position: left calc(50% + 20px); のような式により、 — 背景の 左辺 が中央から 20px の所に位置するのではなく — 正しく，背景が中央に寄せられてから, 右方へ 20px ずらされるようにするため、長さには直接的に解決されないまま，保全される。 Due to this, background-position preserves the percentage in a calc() rather than resolving it directly into a length, so that an expression like background-position: left calc(50% + 20px) properly centers the background and then shifts it 20px to the right, rather than placing its left edge 20px off of center.

テーブルのセルや要素の 横幅／縦幅 の計算は複雑なので、テーブルの［ 列, 列グループ, 行, 行グループ, セル ］の横幅や縦幅に対する百分率が含まれる数式については、テーブル­レイアウトが自動（ auto ）か固定（ fixed ）かに関わらず， auto が指定されていたかのように 扱われてもよい。 Given the complexities of width and height calculations on table cells and table elements, math expressions involving percentages for widths and heights on table columns, table column groups, table rows, table row groups, and table cells in both auto and fixed layout tables MAY be treated as if auto had been specified.

8.1.4. 範囲検査

構文解析時には、 calc() の中の値に対しては範囲検査は遂行されないので、値が範囲外であっても宣言は無効にならない。 しかしながら、式による結果の使用値は，当の文脈に許容される範囲に切り詰められなければならない。 Parse-time range-checking of values is not performed within calc(), and therefore out-of-range values do not cause the declaration to become invalid. However, the used value resulting from an expression must be clamped to the range allowed in the target context.

注記： したがって、 calc() を受容するすべての文脈において，許容される値の範囲は（開区間ではなく）閉区間として定義することが要求される。 Note: This requires all contexts accepting calc() to define their allowable values as a closed (not open) interval.

0px より小さい横幅は許容されないので、次の 3 つは width: 0px; に等価になる： Since widths smaller than 0px are not allowed, these three declarations are equivalent: 

width: calc(5px - 10px);
width: calc(-5px);
width: 0px;

しかしながら、 width: -5px; と width: calc(-5px); は等価ではないことに注意。 calc() の外側にある範囲外の値は，構文的に無効であり、宣言はまるごと落とされることになる。 Note however that width: -5px is not equivalent to width: calc(-5px)! Out-of-range values outside calc() are synactically invalid, and cause the entire declaration to be dropped.

8.1.5. 直列化

calc() 値を直列化するときは： To serialize a calc() value:

  1. 式を次にしたがって単純化する： Simplify the expression by:

    1. 入れ子にされた calc() 値を，その引数を丸括弧で括ったものに置換する。 Replacing nested calc() values with parentheses containing their contents
    2. すべての乗算／除算を解決する。 Resolving all multiplications and divisions
    3. 単位が一致するものどうしは結合する。 Combining identical units

    注記： 値の算出­処理により、異質な単位は一致するように変形される。 例えば em と px は，指定値の時点では はっきり異なるが、算出値の時点では，いずれも px 単位に絶対化される。 Note: The value-computation process can transform disparate units into identical ones. For example, em and px are obviously different at specified-value time, but at computed-value time they’re both absolutized to px.

    結果は、一意な単位たちの総和にならなければならない。 （この総和においては、値ゼロの項も保全されなければならない。） The result must be a summation of unique units. (Terms with a value of zero must be preserved in this summation.)

  2. 前段による単純化­処理の結果が単独の値になる（ 1 個の［ <number> ／ <dimension> ／ <percentage> ］）, かつ 直列化されている値は［ 算出値以降の値 ］である場合、［ 括っている calc() を外した その 1 個の値 ］として直列化する。 値が当の文脈において範囲外になる場合、許容されるもっとも近傍の値に切り詰められなければならない。 If this simplification process results in only a single value (one <number>, one <dimension>, or one <percentage>), and the value being serialized is a computed value or later, serialize it just as that one value, without the calc() wrapper. If this value is outside the allowed range for the context, it must be clamped to the nearest allowed value.
  3. 他の場合、その総和を包含している calc() として直列化する — 各 単位は ASCII 文字大小無視でアルファベット順に並べ， 実数は どの単位よりも前に, 百分率は どの単位よりも後に来るようにする。 Otherwise, serialize as a calc() containing the summation, with the units ordered ASCII case-insensitive alphabetically, the number (if present) coming before all units, and the percentage (if present) coming after all units.

例えば， calc(20px + 30px) は、指定値としては calc(50px) に，算出値としては 50px に直列化する。 For example, calc(20px + 30px) would serialize as calc(50px) as a specified value, or as 50px as a computed value.

calc(20px + 0%) の様な値は、 calc(20px + 0%) として直列化し，いずれの項も 直列化された値の中に維持する。 （ゼロ値の項を維持することは重要である — いずれかの値が一時的にゼロ値になっても，推移の最中に突然 calc() の “形が変わる” ことがないように。 また、すべての項がゼロであるときに “単位を選ぶ” 必要もなくなる。） A value like calc(20px + 0%) would serialize as calc(20px + 0%), maintaining both terms in the serialized value. (It’s important to maintain zero-valued terms, so the calc() doesn’t suddenly "change shape" in the middle of a transition when one of the values happens to have a zero value temporarily. This also removes the need to "pick a unit" when all the terms are zero.)

calc(20px + 2em) の様な値は、指定値としては calc(2em + 20px) に直列化する（両 単位とも、指定値の時点では互換でないので，アルファベット順に整列した上で維持する）。 また、算出値としては 52px のような値に直列化する（ em 値は 算出値の時点で絶対­長さに変換されるので、 1em = 16px であったなら、それらは 52px に結合した上で，括っている calc() を外す） A value like calc(20px + 2em) would serialize as calc(2em + 20px) as a specified value (maintaining both units as they’re incompatible at specified-value time, but sorting them alphabetically), or as something like 52px as a computed value (em values are converted to absolute lengths at computed-value time, so assuming 1em = 16px, they combine into 52px, which then drops the calc() wrapper.)

8.2. 一連の値のトグル ： toggle()

toggle() 式は、子孫の一連の要素が，同じ値を継承する代わりに，値のリストから循環的に値を得られるようにする。 The toggle() expression allows descendant elements to cycle over a list of values instead of inheriting the same value.

次の例は、 <em> 要素を一般の場所では斜字体にしつつ、斜字体の内側では通常体に戻す： The following example makes <em> elements italic in general, but makes them normal if they’re inside something that’s italic: 

em { font-style: toggle(italic, normal); }

次の例は、入れ子にされたリストのマーカを順繰りに循環させる。 トップレベルのリストのマーカは disc 形状にされ、 その中に入れ子にされたリストのマーカは，階層が深まる度に順に circle, square, box にされ, （ 5 階層の深さで）また disc から始まるようになる。 The following example cycles markers for nested lists, so that a top level list has disc-shaped markers, but nested lists use circle, then square, then box, and then repeat through the list of marker shapes, starting again (for the 5th list deep) with disc. 

ul { list-style-type: toggle(disc, circle, square, box); }

toggle() 式の構文は次で与えられる： The syntax of the toggle() expression is: 

toggle( <toggle-value># )

ここで <toggle-value> は、式が置かれる場所で妥当かつ, トップレベルのカンマは含まないような，任意の CSS 値である。 内側のいずれかの値が妥当でない場合、 toggle() 式­全体が無効になる。 toggle() 式は、任意のプロパティの値に利用してよいが，そのプロパティ値の唯一の成分でなければならない。 where <toggle-value> is any CSS value that is valid where the expression is placed, and that doesn’t contain any top-level commas. If any of the values inside are not valid, then the entire toggle() expression is invalid. The toggle() expression may be used as the value of any property, but must be the only component in that property’s value.

toggle() 記法は入れ子にできない。 また attr() や calc() も含められない。 そのような構成子を含む宣言は無効になる。 The toggle() notation is not allowed to be nested; nor may it contain attr() or calc() notations. Declarations containing such constructs are invalid.

次の toggle() 式の例は、どれも無効である： The following toggle() examples are all invalid: 

background-position: 10px toggle(50px, 100px);
/* toggle() 式はプロパティの唯一の成分でなければならない */

list-style-type: toggle(disc, 50px);
/* 50px は list-style-type の妥当な値でない */

toggle() 式の算出値を決定するためには、最初に それぞれの引数を，それが式が置かれたプロパティの唯一の成分であったかのように評価して，引数ごとに表現される算出値を得る — n 番目の引数に対し得られた値を Cn とする。 次に、各 Cn をプロパティの継承値 と比較する。 toggle() の算出値は、［ Cn が継承値に合致するような最小の n ］に対する Cn+1 になる。 n が最後の引数に対応する場合, あるいは合致する n が無い場合の算出値は、最初の引数で表現される算出値 C1 になる。 To determine the computed value of toggle(), first evaluate each argument as if it were the sole value of the property in which toggle() is placed to determine the computed value that each represents, called Cn for the n-th argument to toggle(). Then, compare the property’s inherited value with each Cn. For the earliest Cn that matches the inherited value, the computed value of toggle() is Cn+1. If the match was the last argument in the list, or there was no match, the computed value of toggle() is the computed value that the first argument represents.

注記： したがって， toggle() 内で同じ値が繰り返された場合、リストは短絡されることになる。 例えば toggle(1em, 2em, 1em, 4em) は toggle(1em, 2em) に等価になる。 Note: This means that repeating values in a toggle() short-circuits the list. For example toggle(1em, 2em, 1em, 4em) will be equivalent to toggle(1em, 2em).

注記： toggle() は明示的に親の算出値を見るので、非継承プロパティでも機能する。 これは、非継承プロパティにおける inherit キーワードと同様なふるまいになる。 Note: That toggle() explicitly looks at the computed value of the parent, so it works even on non-inherited properties. This is similar to the inherit keyword, which works even on non-inherited properties.

注記： プロパティの算出値は抽象的な値の集合であり、特定の方法による直列化 [CSS21] で与えられるものではない。 従って、算出値の比較は常に一義的に定まり, 予期される結果になる筈である。 例えば，Level 2 の background-position の算出値は、それぞれが絶対­長さまたは百分率として表現される，ちょうど 2 個のオフセットを持つので、宣言 background-position: top center; および background-position: 50% 0%; が生成する算出値は，一致する。 プロパティ定義の “算出値” の欄が多義的, 厳格過ぎに思われる場合は、修正に取り掛かれるよう フィードバック を願う。 Note: That the computed value of a property is an abstract set of values, not a particular serialization [CSS21], so comparison between computed values should always be unambiguous and have the expected result. For example, a Level 2 background-position computed value is just two offsets, each represented as an absolute length or a percentage, so the declarations background-position: top center and background-position: 50% 0% produce identical computed values. If the "Computed Value" line of a property definition seems to define something ambiguous or overly strict, please provide feedback so we can fix it.

toggle() が簡略形プロパティに利用された場合、その各 冗長形下位プロパティの値は， toggle() 式であって, その一連の引数が［ 元の toggle() 式の引数のそれぞれについて，それが その簡略形の唯一の成分であったとするときに，下位プロパティが受け取ることになる値 ］にされたものになる。 If toggle() is used on a shorthand property, it sets each of its longhands to a toggle() value with arguments corresponding to what the longhand would have recieved had each of the original toggle() arguments been the sole value of the shorthand.

例えば、次の簡略形による宣言は： For example, the following shorthand declaration: 

margin: toggle(1px 2px, 3px 4px, 1px 5px);

次の冗長形による宣言に等価になる： is equivalent to the following longhand declarations: 

margin-top:    toggle(1px, 3px, 1px);
margin-right:  toggle(2px, 4px, 5px);
margin-bottom: toggle(1px, 3px, 1px);
margin-left:   toggle(2px, 4px, 5px);

top ／ bottom マージンでは 1px が 2 度現れているので， 2 個の値のみを行き来する一方、 left ／ right マージンでは 3 個の値が順繰りに利用されることになる。 言い換えれば、上の宣言の算出値は，次の冗長形による宣言と同じになる： Note that, since 1px appears twice in the top and bottom margins, they will cycle between only two values while the left and right margins cycle through three. In other words, the declarations above will yield the same computed values as the longhand declarations below: 

margin-top:    toggle(1px, 3px);
margin-right:  toggle(2px, 4px, 5px);
margin-bottom: toggle(1px, 3px);
margin-left:   toggle(2px, 4px, 5px);

その結果は意図されるものではないであろう。 which may not be what was intended.

8.3. 属性­参照： attr()

要素や疑似要素に適用されるプロパティの成分値には、 attr() 関数が許容される。 これは要素の属性の値を返す。 疑似要素に利用された場合、その出自要素の属性の値を返す。 The attr() function is allowed as a component value in properties applied to an element or pseudo-element. It returns the value of an attribute on the element. If used on a pseudo-element, it returns the value of the attribute on the pseudo-element’s originating element.

【 疑似要素 — pseudo-element 】【 属性 — 要素の内容属性（ content attribute ）を指す（ DOM Element の IDL 属性ではなく） 】【 出自要素 — originating element 】

attr() 式の算出値は、下の規則に従って，要素­上の［ 指定された名前の属性 ］の値になる。 The computed value of the attr() expression is the value of the attribute with the specified name on the element, according to the rules given below.

注記： CSS2.1 [CSS21] の下では、 attr() 式は，常に文字列を返す。 CSS3 の下では、 attr() 式は，多様な型の値を返し得る。 が、何でも返せるわけではない。 例えば，［ カウンタ†, 文字列­名†, 引用符†, （ auto, nowrap, baseline などの）キーワード値 ］などのような値は返せない。 これは意図的にそのようにされている。 attr() 式の目的が、 CSS を用いて言語の呈示上の整形法を記述するためではなく， CSS に意味論的なデータを織り込めるようにする所にあるからである。 【 † — おそらく順に counter, named string, quotes を指す。 】 Note: In CSS2.1 [CSS21], the attr() expression always returns a string. In CSS3, the attr() expression can return many different types. The attr() expression cannot return everything, for example it cannot do counters, named strings, quotes, or keyword values such as auto, nowrap, or baseline. This is intentional, as the intent of the attr() expression is not to make it possible to describe a presentational language’s formatting using CSS, but to enable CSS to take semantic data into account.

attr() 式に対する新たな構文は、次で与えられる： The new syntax for the attr() expression is: 

attr( <attr-name> <type-or-unit>? [ , <attr-fallback> ]? )

ここで：

  • <attr-name> 引数は、 CSS 有修飾名（ [CSS3NAMESPACE] の qname 生成規則）であり，属性の名前を表現する。 （名前空間が不在ならば、これはちょうど CSS 識別子になる。） where <attr-name> is a CSS qualified name (the qname production in [CSS3NAMESPACE]) that represents an attribute name. (In the absence of namespacing, this will just be a CSS identifier.)
  • 任意選択の <type-or-unit> 引数は、属性値の解釈の仕方を UA に指示するものであり，後述のキーワード値のいずれかをとる。 それは attr() 式の型を定義する。 省略された場合、暗黙的に string が指定される。 The optional <type-or-unit> argument is a keyword drawn from the list below that tells the UA how to interpret the attribute value, and defines a type for the attr() expression. If omitted, string is implied.

  • 任意選択の <attr-fallback> 引数は、次のいずれかの場合に利用される，フォールバック値を表現する：

    • 要素にその名前の属性が無い
    • 属性の値は与えられた型に解析できない
    • 属性の値は当のプロパティ値として無効であるか または範囲外にある

    <attr-fallback> の不在の下では、後述の <type-or-unit> の既定値が暗黙的に指定される。

    The optional <attr-fallback> argument represents a fallback value, which is used if the named attribute is missing, or its value cannot be parsed into the given type or is invalid/out-of-range for the property. If it’s absent, the default value for the given <type-or-unit> (from the list below) is implied.

注記： toggle() の <toggle-value> 引数と違って、 attr() の <attr-fallback> 引数は 常に関数記法の最後の引数になるので，トップレベルにもカンマを含ませられる。 Note: Unlike <toggle-value>s, an attr() <attr-fallback> value may contain top-level commas, as it is always the last argument in the functional notation.

attr() 式は、次のすべてが成立する場合に限り妥当である： The attr() expression is only valid if:

  • attr() 式の型は attr() 式が置かれる場所で妥当である。 the attr() expression’s type is valid where the attr() expression is placed,
  • 名前空間­接頭辞を伴う属性­名に対しては、その接頭辞が 【 名前空間­宣言により 】 定義されている。 the namespace prefix of the attribute name, if any, is defined,
  • <attr-fallback> は attr() 式が置かれる場所で妥当である。 the <attr-fallback> is valid where the attr() expression is placed,
  • <attr-fallback> は別の attr() 式を含んでいない。 the <attr-fallback> does not contain another attr() expression,
  • attr() 式がプロパティの唯一の成分値でない場合は、 <attr-fallback> は attr() の型に合致している。 and, if the attr() expression is not the sole component value of a property, the <attr-fallback> matches the attr()’s type

注記： attr() 式がプロパティ値の全体である場合、既定値（フォールバック値）は与えられた型である必要はない。 例えば，作者が要する属性の型が px である場合の既定は、依然として auto も可能である。 width: attr(size px, auto); のように。 Note that the default value need not be of the type given, if the attr() expression is the entire property value. For instance, if the type required of the attribute by the author is px, the default could still be auto, like in width: attr(size px, auto);.

一方で、 attr() 式が全部的なプロパティ値を形成するために他の値と併用される場合の既定値は， attr() の型に合致しなければならない。 例えば， box-shadow: attr(size px, inset) 5px 10px blue; は、 attr() 式が［ px による長さ, または inset キーワード ］のいずれに 置換されても，妥当な宣言になるが、それでも無効である。 If the attr() is used alongside other values to form the full property value, however, then the default value must match the attr()'s type. For example, box-shadow: attr(size px, inset) 5px 10px blue; is invalid, even though it would create a valid declaration if you substituted the attr() expression with either a px length or the inset keyword.

指定された属性が要素­上に存在する場合、（以下のリストにて定義される） <type-or-unit> 引数の要件に従って，属性の値が構文解析されなければならない。 型が string である場合を除き、最初に，先頭と末尾の空白は取り除かれなければならない。 結果の値が attr() 式の値になる。 値が要件を満たす形に構文解析できない場合、フォールバック値が attr() 式の値になる。 If the specified attribute exists on the element, the value of the attribute must be parsed as required by the <type-or-unit> argument (as defined in the list below). Unless the type is string, it must first be stripped of leading and trailing white space. The resulting value is the attr() expression’s value. If the value did not parse as required, the attr() expression’s value is its fallback value.
