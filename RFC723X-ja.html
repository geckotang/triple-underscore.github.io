Hypertext Transfer Protocol (HTTP/1.1)

要約

~HTTP（ Hypertext Transfer Protocol ）は、分散型の協調的な~hypertext情報~systemのための、`~stateless$な~app-levelの~protocolである。 この仕様は、次の文書から構成される： ◎ The Hypertext Transfer Protocol (HTTP) is a stateless application-level protocol for distributed, collaborative, hypertext information systems.

  • RFC7230 — ~messageの構文と経路制御 （ Message Syntax and Routing ）
  • RFC7231 — 意味論と内容（ Semantics and Content ）
  • RFC7232 — 条件付き要請（ Conditional Requests ）
  • RFC7233 — 範囲~要請（ Range Requests ）
  • RFC7234 — ~caching（ Caching ）
  • RFC7235 — 認証（ Authentication ）

この頁では、上述の各種 仕様の共通する部分を集約する。

加えて、編集の都合により、これらの仕様の総集的 ABNF （ Appendix ）を， 別~頁に分離している。

    構文の表記法

    この仕様は `5234$R による ABNF（ Augmented Backus-Naur Form ）表記法を利用する。 また、 `7230-7$rfcにて定義される~list拡張を利用する — それは、（ "`*^c" 演算子が指示する繰返しに類似した）`~list演算子$（ "`#^c" ）を利用して，~comma区切りの~listを簡潔に定義できるようにする。 ◎ This specification uses the Augmented Backus-Naur Form (ABNF) notation of [RFC5234] with a list extension, defined in Section 7 of [RFC7230], that allows for compact definition of comma-separated lists using a '#' operator (similar to how the '*' operator indicates repetition).

    この仕様にて利用される用語： “文字”, “文字~符号化スキーム”, “~charset”, “`~protocol要素@†” は、`6365$Rにて定義される。 ◎ This specification uses the terms "character", "character encoding scheme", "charset", and "protocol element" as they are defined in [RFC6365].

    【† 単に “要素” とも記される。 `6365-6$rfc から定義を引用： `~protocol要素$ とは、一意的に命名された，~protocolの部品である。 ◎ Protocol elements are uniquely named parts of a protocol. 】

    日本語への翻訳に関する注記

    RFC 原文は，ほぼ素のテキストであり、この日本語訳では、 HTML マークアップ ／ CSS スタイル の利用により，記述が等価になる所では、一部の対訳を省略している。 例えば：

    • 用語の定義箇所を示す，節 番号への参照の多くは、代わりに用語そのものにリンクをあてがった上で，節 番号の記述を省略している。
    • リンク先の説明で代用になるときには、記述を省くことがある。
    • 一部の語 — 例えば “状態°コード（ status code ）” は、コード番号が記されている所では、（その番号に専用のスタイルがあてがわれるため，）自明なので省略することもある。

    また、見通しを良くするため、原文のかなりの割合の段落は，マークアップを利用してより明示的な文脈階層に細分化している。 例えば：

    • 構造を細分化すると同時に，複数の文から共有されている語句（例えば主語）, あるいは文脈を指示している記述を一箇所に集約している所もある。
    • 同じ文脈に属する複数の段落を，リスト化などにより一つのブロック階層にまとめている所もある。
    • 文や語句の互いの論理的なつながりや関係性［ を文脈から推し量る必要がある ／ がそのまま日本語訳すると不明瞭になる ］所では、階層化により明確化している。

    原文には、（特に同じ段落内で，）同じ対象を，異なる［ 言い回し, または語句 ］により言及している箇所も多数見られる。 紛らわしいので、細分化に際しては，そのような言及もなるべく一つに［ 集約, あるいは結合 ］している（例えば、 明示的な指示代名詞マークアップ などを用いて — 日本語には “the” が無い）。

    以上に述べたようなマークアップは，訳者による解釈も含むので、解釈に誤りがあれば，それを反映することになる（例えば、原文の語句（例えば “host” ）が、構文の構成子か, 用語か, 意味の記述か, いずれを表すのか、あるいはそれらを兼ねているとしても（多くの場合，そうである），いずれが主として 意図されている／念頭に置かれているかは、必ずしも自明ではない）。

    一部の語の対訳は，“忠実さに欠く” 語, あるいは一般に見受けられる語と異なる語に代えている。 例えば：

    • 語 “header field” は、一律に “ヘッダ（ header ）” と略して対訳している。 “header field” の略記として用いられている語 “field” も同様に置換している。 “ヘッダ” が他の意味で用いられる所は無く， “フィールド” より “ヘッダ” の方が一般に通りがよいと見受けられるので。
    • “回送（ forward ）” ： “転送” あるいは “自動転送” という対訳もあるが、前者は “transfer” の対訳と衝突し，後者は長い（また， “automatic な transfer” と誤解され易くもなる）ので。
    • “条態（ condition ）” ： これは、訳者による造語（条件＋状態）である。 “状況／状態” , あるいは “条件” と訳されることが多いが、前者は “situation”, “circumstance” ／ “state” などの対訳と衝突し，後者は文脈にそぐわないことが多々あるので（無論、 “コンディション” とも訳せるが，用語 “条件付き（ conditional ）” との関連性もあるので）。
    • “全部的（ full ）” — “完全（ complete ）” との区別を要するため。 また、 “部分的（ partial ）” との対比も兼ねるため。
    • “検証°（ verify ）” （正しく作られたかどうかを検証）— “検証（ validate ）” （作られたものが正しいかどうかを検証）と区別するため。
    • “状態°（ status ）” （状態の集合／状態を［指示する／識別する］もの） — “状態（ state ）” （ある特定の状態）と区別するため。

    ウィンドウ下端には、対訳の切り替え機能が用意されている。
