1. 序論

CSS の根底をなす設計原理の一つにカスケード法（カスケード処理規則, cascading ）がある。 これにより、複数のスタイルシートを文書の呈示に作用させられるようになる。 同じ ［要素, プロパティ］ の組み合せに対し，複数の宣言が与えられている下での値の競合は、何らかの方法で解決される必要がある。 One of the fundamental design principles of CSS is cascading, which allows several style sheets to influence the presentation of a document. When different declarations try to set a value for the same element/property combination, the conflicts must somehow be resolved.

逆に、 ［要素, プロパティ］ の組み合せに値を設定する宣言が存在しない場合についても，解決される必要がある。 この場合、値は，継承やプロパティの初期値を通して見出される。 The opposite problem arises when no declarations try to set a the value for an element/property combination. In this case, a value is be found by way of inheritance or by looking at the property’s initial value.

カスケード法とデフォルト法を経ることにより，入力に与えられた一連の宣言から、各­要素の各プロパティに対する指定値が出力される。 The cascading and defaulting process takes a set of declarations as input, and outputs a specified value for each property on each element.

この仕様は、文書­内のすべての要素­上のすべてのプロパティについて，それぞれの指定値を見出すための規則を述べる。 ページ文脈およびそのマージン­ボックスの下で，指定値を見出すための規則は、 [CSS3PAGE] にて述べられる。 The rules for finding the specified value for all properties on all elements in the document are described in this specification. The rules for finding the specified values in the page context and its margin boxes are described in [CSS3PAGE].

このモジュールは、 [CSS21] の 6 章にて定義される［ プロパティ値のあてがわれ方, カスケード処理, 継承 ］のための規則を置換し, それを拡張する。 This module replaces and extends the rules for assigning property values, cascading, and inheritance defined in [CSS21] chapter 6.

For example, the Media Queries Level 4 specification, when combined with this module, expands the definition of the <media-query> value type as used in this specification.

2. スタイルシートの import 処理： @import 規則

@import 規則により，利用者は 他のスタイルシートからスタイル規則を import （取り込む）できるようになる。 @import 規則が妥当なスタイルシートを指しているならば、 UA は、次の二つの例外を除き， @import 規則の場所にその内容が記されていたかの様に取り扱わなければならない： The @import rule allows users to import style rules from other style sheets. If an @import rule refers to a valid stylesheet, user agents must treat the contents of the stylesheet as if they were written in place of the @import rule, with two exceptions:

  • ある特色機能が，適用されるのは［ ある特定のスタイルシートのみであり， import されたそれには適用されない ］ものと明示的に定義している場合（ @namespace 規則など）、それは import されたスタイルシートには適用されない。 If a feature explicitly defines that it only applies to a particular stylesheet, and not any imported ones (such as the @namespace rule), then it doesn’t apply to the imported stylesheet.
  • ある特色機能が，スタイルシート内の複数の構成子に相対的な位置に依拠する場合（ @else 規則は，別の条件付きグループ規則に後続するものとする要件や、 @charset 規則には，他の内容が先行してはならないとする要件など）、それが適用されるのは，同じスタイルシート内の構成子たちの合間に限られる。 If a feature relies on the relative position of two or more constructs in a stylesheet (such as the requirement that an @else rule follow another conditional group rule, or the requirement that @charset must not have any other content preceding it), it only applies between constructs in the same stylesheet.

例えば，［ import されたスタイルシートからのスタイル規則 ］の中の宣言は、それがスタイルシートの中の @import の地点にそのままの形で記されたかのように，カスケードに対し作用する。 同様に，［ 視野が絞られたスタイルシートの中に import されたスタイルシート ］内のスタイル規則は、同じように視野が絞られる。 For example, declarations in style rules from imported stylesheets interact with the cascade as if they were written literally into the stylesheet at the point of the @import. Similarly, style rules in a stylesheet imported into a scoped stylesheet are scoped in the same way.

スタイルシート内では、どの @import 規則も， @charset を除く 他のすべての at-規則／スタイル規則よりも，前に位置していなければならない。 （ @charset については，存在するならばスタイルシートの先頭に位置していなければならない）。 さもなければ @import 規則は無効とされる。 @import の構文は次で与えられる： Any @import rules must precede all other at-rules and style rules in a style sheet (besides @charset, which must be the first thing in the style sheet if it exists), or else the @import rule is invalid. The syntax of @import is: 

@import [ <url> | <string> ]
   [ supports( [ <supports-condition> | <declaration> ] ) ]?
   <media-query-list>? ;

<url> ／ <string> は、 import されることになるスタイルシートの URL を与える。 import 条件 と総称される，任意選択の［ [ <supports-condition> | <declaration> ], および <media-query-list> ］は、そのスタイルシートが どのような条件の下で適用されるかを定める。 where the <url> or <string> gives the URL of the style sheet to be imported, and the optional [<supports-condition>|<declaration>] and <media-query-list> (collectively, the import conditions ) state the conditions under which it applies.

次の 条件付き @import 規則は、 UA が display: flex をサポートする場合にのみスタイルシートを読み込み， 表示域の横幅が 400px 以下の 携帯機器（ handheld ）に対してのみスタイルシートを適用する： The following conditional @import rule only loads the style sheet when the UA supports display: flex, and only applies the style sheet on a handheld device with a maximum viewport width of 400px. 

@import url("narrow.css") supports(display: flex) handheld and (max-width: 400px);

<string> が供されている場合、それは，同じ引数を伴う１個の <url> として解釈されなければならない。 If a <string> is provided, it must be interpreted as a <url> with the same value.

次の２行（ url() によるものと, 文字列そのままのもの）は、意味的に等価な @import 構文になる： The following lines are equivalent in meaning and illustrate both @import syntaxes (one with url() and one with a bare string): 

@import "mystyle.css";
@import url("mystyle.css");

2.1. 条件付き @import 規則

import は， import 条件により，［ 媒体依存／特色機能サポート依存 ］にすることもできる。 import 条件 が不在の下では，無条件に import される（その <media-query-list> に対する値 all の指定も，同じ効果になる）。 import 条件が合致しない場合、 import されるスタイルシート内の規則は適用されない — import されたスタイルシートが，与えられた媒体クエリを伴う［ @media ／ @supports ］ブロックで括られたときとちょうど同じ様に。 The import conditions allow the import to be media– or feature-support–dependent. In the absence of any import conditions, the import is unconditional. (Specifying all for the <media-query-list> has the same effect.) If the import conditions do not match, the rules in the imported stylesheet do not apply, exactly as if the imported stylesheet were wrapped in @media and/or @supports blocks with the given conditions.

@import 規則を媒体依存に仕立てる例を以下に示す： The following rules illustrate how @import rules can be made media-dependent: 

@import url("fineprint.css") print;
@import url("bluish.css") projection, tv;
@import url("narrow.css") handheld and (max-width: 400px);

UA は従って， import 条件が合致しない条件付き import については、リソースへ取得をかけずに済ませられる。 加えて、 import によるスタイルシートの適用が <supports-condition> により阻まれる場合には、そのスタイルシートに対し，（他のリンクを通して読み込まれるのでない限り）取得をかけてはならず、その import 規則に対応する CSSImportRule.styleSheet 値は，（他のリンクを通して読み込まれる場合でも） null を返さなければならない。 User agents may therefore avoid fetching a conditional import as long as the import conditions do not match. Additionally, if a <supports-condition> blocks the application of the imported style sheet, the UA must not fetch the style sheet (unless it is loaded through some other link) and must return null for the import rule’s CSSImportRule.styleSheet value (even if it is loaded through some other link).

作者が、より新しい UA のネットワーク処理能に響かせることなく，旧来の UA にフォールバック規則を提供する方法を，次の規則に示す： The following rule illustrates how an author can provide fallback rules for legacy user agents without impacting network performance on newer user agents: 

@import url("fallback-layout.css") supports(not (display: flex));
@supports (display: flex) {
  ...
}

<media-query> は、 media_query_list 生成規則に対応し，媒体クエリとして解釈される。 <supports-condition> は、 supports_condition 生成規則に対応し， @supports 条件として解釈される。 <supports-condition> に代わって <declaration> （ declaration 生成規則） が与えられた場合、 supports_declaration_condition 生成規則として解釈された上で（すなわち，追加の丸括弧を要する）， <supports-condition> として扱われなければならない。 A <media-query> corresponds to the media_query_list production and is interpreted as a media query, and a <supports-condition> corresponds to a supports_condition production and is interpreted as an @supports condition. If a <declaration> (a declaration production) is given in place of a <supports-condition>, it must be interpreted as a supports_declaration_condition production (i.e. the extra set of parentheses is implied) and treated as a <supports-condition>.

例えば、次の二行は等価になる： For example, the following two lines are equivalent: 

@import "mystyle.css" supports(display: flex);
@import "mystyle.css" supports((display: flex));

import 条件の評価とその全部的な構文は Media Queries [MEDIAQ], および CSS Conditional Rules [CSS3-CONDITIONAL] にて規定される。 The evaluation and full syntax of the import conditions are defined by the Media Queries [MEDIAQ] and CSS Conditional Rules [CSS3-CONDITIONAL] specifications.

2.2. import によるスタイルシートの処理

同じスタイルシートが文書­内の複数の場所から import またはリンクされている場合、 UA は，それぞれのリンクが独立するスタイルシートを指しているかのように，ふるまわなければならない。 When the same style sheet is imported or linked to a document in multiple places, user agents must process (or act as though they do) each link as though the link were to an independent style sheet.

注記： これはリソース取得に要件を課すものではない。 スタイルシートが CSSOM にどのように反映されるかや、この仕様の様に，他の仕様の中でどのように利用されるかについての要件になる。 複数の場所でリンクされている, あるいは複数回 import されたとしても，適切なキャッシュ処理の下で 一度だけスタイルシートへ取得をかけることは、ごく適切な UA のふるまいになる。 Note: This does not place any requirements on resource fetching, only how the style sheet is reflected in the CSSOM and used in specs such as this one. Assuming appropriate caching, it is perfectly appropriate for a UA to fetch a style sheet only once, even though it’s linked or imported multiple times.

import されたスタイルシートの出自は、それを import したスタイルシートの出自と同じになる。 The origin of an imported style sheet is the origin of the style sheet that imported it.

import されたスタイルシートの 環境­符号化方式 は、それを import したスタイルシートの符号化方式で与えられる。 [CSS3SYN] The environment encoding of an imported style sheet is the encoding of the style sheet that imported it. [CSS3SYN]

2.3. CSS スタイルシートの Content-Type

import されたスタイルシートの処理は、リンク先のリソースの実際の種別に依存する — 種別は、次にしたがって決定される：

  • リソースに Content-Type メタデータ がない場合の種別は、 text/css 。
  • ホスト文書が過去互換モード下にあって, かつ import されたスタイルシートと同一生成元である場合の種別も、 text/css 。
  • 他の場合の種別は、リソースの Content-Type メタデータ から決定される。
The processing of imported style sheets depends on the actual type of the linked resource. If the resource does not have Content-Type metadata, or the host document is in quirks mode and has the same origin as the imported style sheet, the type of the linked resource is text/css. Otherwise, the type is determined from its Content-Type metadata.

リンク先のリソースの種別が text/css である場合、 CSS スタイルシートとして解釈されなければならない。 他の場合、ネットワークエラーとして解釈されなければならない。 If the linked resource’s type is text/css, it must be interpreted as a CSS style sheet. Otherwise, it must be interpreted as a network error.

3. 簡略形プロパティ

作者が種々のプロパティの値を単独のプロパティで一括して指定できるようにするため，一部のプロパティは 簡略形プロパティ （ “shorthand” ）として定められている。 簡略形プロパティは、そのすべての （冗長形）下位プロパティ （ “longhand sub-property” または単に “longhand” ）を，その場で展開されたかのように設定する。 Some properties are shorthand properties, meaning that they allow authors to specify the values of several properties with a single property. A shorthand property sets all of its longhand sub-properties, exactly as if expanded in place.

簡略形プロパティから値が省略されて “消失した” それぞれの下位プロパティには、他から定義されない限り，その初期値があてがわれる。 When values are omitted from a shorthand form, unless otherwise defined, each “missing” sub-property is assigned its initial value.

注記： すなわち，簡略形プロパティ宣言は常に、そのすべての 下位プロパティを設定する — それらが明示的に設定されなくても。 不用意に簡略形プロパティを利用した場合，一部の下位プロパティを気付かないうちに再設定することになりかねないが、注意して利用すれば，気付かないうちに他のソースからカスケードされた下位プロパティを再設定して，“白紙状態” に戻すことを保証してくれる。 This means that a shorthand property declaration always sets all of its sub-properties, even those that are not explicitly set. Carelessly used, this might result in inadvertently resetting some sub-properties. Carefully used, a shorthand can guarantee a “blank slate” by resetting sub-properties inadvertently cascaded from other sources.

例えば， background-color: green; ではなく background: green; と記すことにより、以前のどこかの宣言で 背景に background-image による画像が設定されているかもしれないときでも，背景色だけにされるように上書きできる。 For example, writing background: green rather than background-color: green ensures that the background color overrides any earlier declarations that might have set the background to an image with background-image.

【 “上書き” — カスケード法における出現順序による順位付けの結果として，そうなる（より高次の順位付けの宣言値（例えばより詳細度が高いもの）まで上書きするわけではない）。 】

例えば，CSS Level 1 の font プロパティは、 font-style, font-variant, font-weight, font-size, line-height, font-family を一括して設定するための，簡略形プロパティである。 したがって，次の様な複数の宣言は： For example, the CSS Level 1 font property is a shorthand property for setting font-style, font-variant, font-weight, font-size, line-height, and font-family all at once. The multiple declarations of this example: 

h1 {
  font-weight: bold;
  font-size: 12pt;
  line-height: 14pt;
  font-family: Helvetica;
  font-variant: normal;
  font-style: normal;
}

次の様に書き直せる can therefore be rewritten as 

h1 { font: bold 12pt/14pt Helvetica }

font に対する新たな下位プロパティが更に導入されれば、簡略形 宣言は，それらも その初期値に再設定することになる。 As more font sub-properties are introduced into CSS, the shorthand declaration resets those to their initial values as well.

【 次の様な場合もあることに注意： border-width のようなプロパティは、簡略形プロパティ border の下位プロパティであると同時に，下位プロパティに border-top-width, border-left-width, … 等々を持つ簡略形プロパティでもある。 この様な場合の border の “下位プロパティ” は、連鎖的に辿った末端の下位プロパティを指しているように見受けられる — が、この場合の “中間” のプロパティ（この例では border-width 等々）が，下位プロパティに［含まれる／含まれない］のどちらに定義されているのかは、はっきりしない — どちらに解釈したとしても，カスケードに現れる結果は変わらないと見受けられるので（末端の下位プロパティに限定する必要がある所では，その旨を付記した方が誤解は少ないであろう）。 あるいは、そうなることを利用して（意図的に曖昧に）定義されているようにも思われる。 実際、何が下位プロパティと見なされるかは，実装にも依存する — 拡張仕様により、それまでは末端のプロパティであったものが，より細分化された下位プロパティを持つ様になることもあるので（例えば overflow プロパティに対する overflow-x, overflow-y ）。 】

一部の簡略形プロパティは、その下位プロパティの値に直接的には対応しないような，異なる構文や特別なキーワードを持ち得る（そのような簡略形プロパティは，その値の展開を明示的に定義することになる）。 In some cases, a shorthand might have different syntax or special keywords that don’t directly correspond to values of its sub-properties. (In such cases, the shorthand will explicitly define the expansion of its values.)

また、簡略形において 再設定のみ になる下位プロパティもある： それは、他の下位プロパティと同様に，簡略形の指定の中で値が指定されていない場合は，初期値に再設定されるが、それ以外の値に設定する構文が，簡略形に備わっていないものである（下記の CSS 全域キーワードは別として）。 例えば、 border 簡略形は， border-image をその初期値 none に再設定するが、それ以外の値に設定する構文を備えていない。 [CSS3BG] In other cases, a property might be a reset-only sub-property of the shorthand: Like other sub-properties, it is reset to its initial value by the shorthand when unspecified, but the shorthand might not include syntax to set the sub-property to any of its other values. For example, the border shorthand resets border-image to its initial value of none, but has no syntax to set it to anything else. [CSS3BG]

簡略形プロパティが CSS 全域キーワード [CSS3VAL] のいずれかとして指定されている場合、そのすべての下位プロパティは，再設定のみのものも含め，そのキーワードに設定される。 （これらのキーワードは、簡略形に限られず，単独の宣言の中で他の値と組み合せることはできないことに注意。） If a shorthand is specified as one of the CSS-wide keywords [CSS3VAL], it sets all of its sub-properties to that keyword, including any that are reset-only sub-properties. (Note that these keywords cannot be combined with other values in a single declaration, not even in a shorthand.)

簡略形プロパティを !important として宣言することは、そのすべての下位プロパティを !important として宣言することに等価になる。 Declaring a shorthand property to be !important is equivalent to declaring all of its sub-properties to be !important.

Note, however, that any "default" style applied to that element (such as, e.g. display: block from the UA style sheet on block elements such as <div>) will also be blown away.

4. 値の処理

UA は、文書の構文解析と文書­木を構築し終えたならば，木­内のどの要素に対しても，および 対応する整形­構造の中のどのボックスに対しても、そのプロパティのうち，目的の媒体型を適用­対象にするものすべてに対し，値をあてがわなければならない。 Once a user agent has parsed a document and constructed a document tree, it must assign, to every element in the tree, and correspondingly to every box in the formatting structure, a value to every property that applies to the target media type.

与えられた要素／ボックスの CSS プロパティの最終的な値は、次の様な複数の段階を経て計算される： The final value of a CSS property for a given element or box is the result of a multi-step calculation:

  • まず最初に、それぞれの要素の各プロパティについて，要素に適用されている，すべての宣言値が収集される。 要素に適用される宣言値はゼロ個以上いくつでもあり得る。 First, all the declared values applied to an element are collected, for each property on each element. There may be zero or many declared values applied to the element.
  • カスケード法により，カスケード値を得る。 カスケード値は、各­要素の各プロパティごとに，高々１個までである。 Cascading yields the cascaded value. There is at most one cascaded value per property per element.
  • デフォルト法により，指定値を得る。 どの要素も，プロパティごとにきっかり１個の指定値を持つ。 Defaulting yields the specified value. Every element has exactly one specified value per property.
  • 値の依存性を解決して，算出値を得る。 どの要素も，プロパティごとにきっかり１個の算出値を持つ。 Resolving value dependencies yields the computed value. Every element has exactly one computed value per property.
  • 文書の整形（ formatting ）を経ることで，使用値を得る。 要素­上のプロパティが使用値を持つのは、そのプロパティが要素に適用し得るときに限られる。 Formatting the document yields the used value. An element only has a used value for a given property if that property applies to the element.
  • 最後に，使用値は、表示­環境による拘束に基づいて，実効値に変換される。 使用値と同様、実効値も，プロパティが要素に適用し得るときにのみ存在する。 Finally, the used value is transformed to the actual value based on constraints of the display environment. As with the used value, there may or may not be an actual value for a given property on an element.

【 プロパティ が 要素に適用し得る とは、そのプロパティが当該の要素の具現化に効果を及ぼし得る — 言い換えれば、その要素は，プロパティの定義表の “適用対象” の欄に挙げられている — ことを意味する。 “プロパティの 個々の宣言 が要素に適用されている” こととは、意味が異なることに注意。 】

4.1. 宣言値

要素に適用されている各プロパティ宣言が、その要素­上のプロパティに対する 宣言値 （ declared value ）を与える。 詳細は宣言の選別法にて。 Each property declaration applied to an element contributes a declared value for that property associated with the element. See Filtering Declarations for details.

これらの値から、カスケードによる処理を経て，１個の “最優先の値” が選定される。 These values are then processed by the cascade to choose a single “winning value”.

4.2. カスケード値

カスケード値 （ cascaded value ）は、カスケードの結果を表現する。 それは、一連の宣言値から，カスケードを経て（カスケードの出力の中で整列されて）選定される値である。 カスケードの出力が空リストである場合、カスケード値は存在しない。 The cascaded value represents the result of the cascade: it is the declared value that wins the cascade (is sorted first in the output of the cascade). If the output of the cascade is an empty list, there is no cascaded value.

4.3. 指定値

与えられたプロパティの 指定値 （ specified value ）は、スタイルシート作者がその要素に意図した値である。 多くの場合、カスケード値が指定値になるが，カスケード値が存在しない場合は、デフォルト法により指定値が導出され，カスケード値に置き換わる。 これにより、どの要素のどのプロパティにも，指定値の存在が保証される。 プロパティのカスケード値が initial ／ inherit 【 加えて， unset ／ revert 】 キーワードであるときには、特別に取り扱われる。 The specified value is the value of a given property that the style sheet authors intended for that element. It is the result of putting the cascaded value through the defaulting processes, guaranteeing that a specified value exists for every property on every element. In many cases, the specified value is the cascaded value. However, if there is no cascaded value at all, the specified value is defaulted. 4.5. 使用値

4.6. 実効値

使用値は、原則として，出力のための利用の準備が整った値であるが、与えられた環境の下では直接的に利用できない可能性もある。 例えば UA は，整数幅の画素境界にしか描画できないかもしれず、その場合，使用値による幅を近似させる必要も生じ得る。 あるいは，要素のフォント­サイズは、フォントの可用性や font-size-adjust プロパティ値に基づいて，調整を要する。 実効値 （ actual value ）とは、使用値に対しその種の調整を施した後の値を意味する。 A used value is in principle ready to be used, but a user agent may not be able to make use of the value in a given environment. For example, a user agent may only be able to render borders with integer pixel widths and may therefore have to approximate the used width. Also, the font size of an element may need adjustment based on the availability of fonts or the value of the font-size-adjust property. The actual value is the used value after any such adjustments have been made.

5. 選別法

宣言値を見出すためには、実装はまず，それぞれの要素ごとに 要素に適用されるすべての宣言を識別しなければならない。 宣言は次 【のすべて】 が成立する場合に 要素に適用される： In order to find the declared values, implementations must first identify all declarations that apply to each element. A declaration applies to an element if:

  • 当該の文書に現在 適用されているスタイルシートに属している。 It belongs to a style sheet that currently applies to this document.
  • 偽に評価される条件付きグループ規則（ conditional group rule ） [CSS3-CONDITIONAL] により除外されていない。 It is not qualified by a conditional rule [CSS3-CONDITIONAL] with a false condition.
  • それが属しているスタイル規則の選択子 [SELECT] が要素に合致する。 （必要に応じて視野の適用（ scoping ）も織り込む。 It belongs to a style rule whose selector matches the element. [SELECT] (Taking scoping into account, if necessary.)
  • 構文として妥当である： すなわち，宣言のプロパティは既知のプロパティ名であり, かつ その宣言の値がそのプロパティの構文に合致している。 It is syntactically valid: the declaration’s property is a known property name, and the declaration’s value matches the syntax for that property.

これらの適用される宣言の値が、各­要素の各プロパティに対する宣言値のリストをなす。 このリストの中の どれが選ばれるかは、次節のカスケード法にて与えられる。 The values of the declarations that apply form, for each property on each element, a list of declared values. The next section, the cascade, prioritizes these lists.

6. カスケード法

カスケード （カスケード処理）は、与えられた要素­上の与えられたプロパティに対する宣言値の，順序の無いリストを入力にとり、以下に従って決定される優先順位により，それらの宣言を整列した上で、 【その中で最も優先順位の高い】 単独のカスケード値を出力する。 The cascade takes a unordered list of declared values for a given property on a given element, sorts them by their declaration’s precedence as determined below, and outputs a single cascaded value.

カスケードにおいては、以下の判定基準に従って，宣言が整列される。 先に示される判定基準が後続の判定基準よりも優先される： The cascade sorts declarations according to the following criteria, in descending order of priority:

出自と重要度（ origin と importance ）

宣言の出自は、文字通りの意味に基づく。 宣言が important になるかどうかは、後述の !important が宣言されているかどうかに基づく。 種々の出自 【と important の組】 の優先順位は、高いものから順に，次で与えられる： The origin of a declaration is based on where it comes from and its importance is whether or not it is declared !important (see below). The precedence of the various origins is, in descending order:

  1. 遷移効果 宣言 [CSS3-TRANSITIONS] Transition declarations [CSS3-TRANSITIONS]
  2. important にされた UA 宣言 Important user agent declarations
  3. important にされた 利用者 宣言 Important user declarations
  4. important にされた 上書き宣言 [DOM-LEVEL-2-STYLE] Important override declarations [DOM-LEVEL-2-STYLE]
  5. important にされた 作者 宣言 Important author declarations
  6. アニメーション 宣言 [CSS3-ANIMATIONS] Animation declarations [CSS3-ANIMATIONS]
  7. 通常の 上書き宣言 [DOM-LEVEL-2-STYLE] Normal override declarations [DOM-LEVEL-2-STYLE]
  8. 通常の 作者 宣言 Normal author declarations
  9. 通常の 利用者 宣言 Normal user declarations
  10. 通常の UA 宣言 Normal user agent declarations

上のリストの中で，より先に挙げた出自に属する宣言が より後に挙げた出自に属する宣言よりも優先される。 Declarations from origins earlier in this list win over declarations from later origins.

【 important の有無により［ ［ 上書き／作者 ］／利用者／ UA ］宣言の優先順位が逆になるが、 上書き宣言と作者 宣言 の順位については， important の有無に関わらず同順になる。 】

視野（ scope ）

宣言は、その影響が 視野ルート要素 （ scoping element ）と呼ばれる要素とその子孫のみに絞り込まれるように，文書の部分木に 視野が絞られる こともある。 例えば， [HTML5] は 視野が絞られた <style> 要素を定義する。 このスタイルシートは、その要素の親（をルートとする部分木）に視野が絞られる。 A declaration can be scoped to a subtree of the document so that it only affects its scoping element and that element’s descendants. For example, [HTML5] defines scoped <style> elements, whose style sheets are scoped to the element’s parent.

２つの宣言の視野ルート要素が先祖／子孫の関係にある場合、通常の規則に対しては，子孫の側の視野ルート要素を持つ宣言が優先される。 important にされた規則に対しては、先祖の側の視野ルート要素を持つ宣言が優先される。 If the scoping elements of two declarations have an ancestor/descendant relationship, then for normal rules the declaration whose scoping element is the descendant wins, and for important rules the declaration whose scoping element is the ancestor wins.

注記： 言い換えれば、通常の宣言（ important にされていない宣言）に対しては，視野が内側になる方の宣言が他方を上書きする一方で， !important 規則に対しては，視野が 外側 になる方の宣言が他方を上書きする。 Note: In other words, for normal declarations the inner scope’s declarations override, but for !important rules outer scope’s override.

この段の目的においては、視野が絞られていない宣言は，文書のルート要素に視野が絞られているものと見なされる。 スタイル属性からの通常の宣言は，その属性を伴う要素に視野が絞られているものと見なされる一方で、 スタイル属性の important 宣言は，文書のルート要素に視野が絞られているものと見なされる。 [CSSSTYLEATTR] For the purpose of this step, all unscoped declarations are considered to be scoped to the root element. Normal declarations from style attributes are considered to be scoped to the element with the attribute, whereas important declarations from style attributes are considered to be scoped to the root element. [CSSSTYLEATTR]

注記： この不規則な取り扱いは、単に［ スタイル属性の詳細度は，他のどの作者­規則よりも高い ］とされていた， CSS Level 1 ／ Level 2 に定義される［ スタイル属性における !important 宣言のふるまい ］に合致させるためである。 [CSS21] Note: This odd handling of !important style attribute declarations is to match the behavior defined in CSS Levels 1 and 2, where style attributes simply have higher specificity than any other author rules. [CSS21]

詳細度（ specificity ）

それぞれの宣言は、それが現れるスタイル規則（の選択子）と同じ詳細度を持つ。 選択子の詳細度の算出方法は、 Selectors モジュール [SELECT] にて述べられている。 この段の目的においては、スタイル規則に属さない宣言（ スタイル属性の内容 など）は、どの選択子よりも高い詳細度を持つものと見なされる。 詳細度が高い宣言ほど，より優先される。 The Selectors module [SELECT] describes how to compute the specificity of a selector. Each declaration has the same specificity as the style rule it appears in. For the purpose of this step, declarations that do not belong to a style rule (such as the contents of a style attribute) are considered to have a specificity higher than any selector. The declaration with the highest specificity wins.

出現順序

文書順 【文書­内（スタイルシート内）に現れる順序】 において後の宣言が，先に現れるものより優先される。 この目的においては： The last declaration in document order wins. For this purpose:

  • import されたスタイルシートからの宣言は、 @import 規則がそのスタイルシートに置き換えられたかの様に，順序­付けられる。 Declarations from imported style sheets are ordered as if their style sheets were substituted in place of the @import rule.
  • 大元の文書から独立にリンクされた複数のスタイルシートからの宣言は、それらのスタイルシートが，ホスト文書­言語により決定されるリンク順で連結されたかの様に扱われる。 Declarations from style sheets independently linked by the originating document are treated as if they were concatenated in linking order, as determined by the host document language.
  • スタイル属性からの宣言は、その スタイル属性が現れた要素の文書順に従って順序­付けられ、どのスタイルシートよりも後に位置する。 Declarations from style attributes are ordered according to the document order of the element the style attribute appears on, and are all placed after any style sheets.

カスケードの出力 は、各­要素­上の各プロパティに対するすべての宣言値を，この節に述べた規則に従って整列したリスト（空もとり得る）である。 The output of the cascade is a (potentially empty) sorted list of declared values for each property on each element.

6.1. カスケード法における出自

各スタイル規則は、それがカスケードのどの段階に取り込まれるかを決定する， 出自 （ cascade origin ）を持つ。 CSS では３種類の基本的な出自が定義されている： Each style rule has an cascade origin, which determines where it enters the cascade. CSS defines three core origins:

作者­出自（ author origin ）
文書­言語の規約に従って，ソース文書に対し作者が指定するスタイルシート。 例えば HTML では，文書­内に含められたり, 外部へリンクされ得る。 The author specifies style sheets for a source document according to the conventions of the document language. For instance, in HTML, style sheets may be included in the document or linked externally.
利用者­出自（ user origin ）
利用者は、個々の文書に対し，スタイル情報を指定し得る。 例えば，利用者がスタイルシートが含まれたファイルを指定することもあれば、 UA が利用者スタイルシート（あるいはそのようにふるまうもの）を生成するインタフェースを提供することもある。 The user may be able to specify style information for a particular document. For example, the user may specify a file that contains a style sheet or the user agent may provide an interface that generates a user style sheet (or behaves as if it did).
UA 出自（ user agent origin ）
適合 UA は、既定のスタイルシート（あるいはそのようにふるまうもの）を適用しなければならない。 UA の既定のスタイルシートは、文書­言語の要素を，文書­言語にて一般に期待されている呈示を満たす様に呈示するべきである（例えば 視覚系ブラウザは， HTML の EM 要素を斜字体フォントで呈示するなど）。 例としては HTML UA スタイルシート [HTML5] がある。 Conforming user agents must apply a default style sheet (or behave as if they did). A user agent’s default style sheet should present the elements of the document language in ways that satisfy general presentation expectations for the document language (e.g., for visual browsers, the EM element in HTML is presented using an italic font). See e.g. the HTML user agent style sheet. [HTML5]

【 この仕様も含む CSS 仕様 全般にわたり、この出自を指して， “〜レベルのスタイル” と記されたり，（ “作者レベルの…” , “UA レベルの…” 等々），更に省略して “利用者スタイル”, 等々と記されることが多い。 】

CSS に対する拡張は、次の追加の出自も定義する： Extensions to CSS define the following additional origins:

上書き出自（ override origin ）
DOM Level 2 Style [DOM-LEVEL-2-STYLE] は、カスケードの中で，作者による他の規則より高い優先レベルに取り込まれる， “上書き” スタイルのためのインタフェースを定義している。 DOM Level 2 Style [DOM-LEVEL-2-STYLE] defines an interface for “override” styles, which enter the cascade at a higher level than other author rules.
アニメーション出自（ animation origin ）
CSS Animations [CSS3-ANIMATIONS] は、実行時にそれらの効果を表現する， “仮想の（ virtual ）” 規則を生成する。 CSS Animations [CSS3-ANIMATIONS] generate “virtual” rules representing their effects when running.
遷移­出自（ transition origin ）
CSS Animations 同様, CSS Transitions [CSS3-TRANSITIONS] も、実行時におけるそれらの効果を表現する， “仮想の” 規則を生成する。 Like CSS Animations, CSS Transitions [CSS3-TRANSITIONS] generate “virtual” rules representing their effects when running.

6.2. important 宣言： !important

CSS では作者と利用者の手によるスタイルシート間の均衡をとることも考慮されている。 既定では、作者スタイルシートの中の規則が，利用者スタイルシートのそれを上書きし、利用者スタイルシートは， UA の既定のスタイルシートのそれを上書きする。 宣言を important にしてカスケードにおける重みを増やすことにより，この優先順位を逆にできる。 CSS attempts to create a balance of power between author and user style sheets. By default, rules in an author’s style sheet override those in a user’s style sheet, which override those in the user-agent’s default style sheet. To balance this, a declaration can be made important, which increases its weight in the cascade and inverts the order of precedence.

[CSS3SYN] にて定義されるように， !important を伴う宣言 — すなわち、その値の中の最後の２つの（空白（ whitespace ）でもコメント（ comment ）でもない）トークンが，［ 区切子トークン "!", 識別子トークン "important" ］の並びである宣言 — は、 important であるとされる。 A declaration is important if it has a !important annotation, as defined by [CSS3SYN]. i.e. if the last two (non-whitespace, non-comment) tokens in its value are the delimiter token ! followed by the identifier token important. 

[hidden] { display: none !important; }

important 宣言は、通常の宣言より優先される。 作者／利用者スタイルシートに !important 宣言が含められてもよい。 利用者 !important 宣言は、作者 !important 宣言を上書きする。 この CSS 特色機能を利用すれば、利用者の手による，呈示­制御のための特別な要件（大きなフォント, 色の組み合せ, 等々）を与えることで、文書のアクセス容易性も向上させられる。 An important declaration takes precedence over a normal declaration. Author and user style sheets may contain !important declarations, with user !important declarations overriding author !important declarations. This CSS feature improves accessibility of documents by giving users with special requirements (large fonts, color combinations, etc.) control over presentation.

important 宣言は、その出自に関わらず，どのアニメーションよりも優先される。 これにより、作者は，重要な事例でアニメート値を上書きできるようになる（アニメート値は通常，他のすべての規則を上書きする）。 [CSS3-ANIMATIONS] Important declarations from all origins take precedence over animations. This allows authors to override animated values in important cases. (Animated values normally override all other rules.) [CSS3-ANIMATIONS]

!important 宣言は UA スタイルシートにも含められてよい。 これらは、作者／利用者によるすべての宣言を上書きする。 User agent style sheets may also contain !important declarations. These override all author and user declarations.

次の例の 利用者スタイルシートの最初の規則は， !important 宣言を含むので、作者スタイルシートの対応する宣言を上書きする。 ２番目の規則の宣言も !important により優先されることになる。 しかしながら，利用者スタイルシートの中の３番目の宣言は !important でないので、作者スタイルシートの中の（簡略形プロパティのスタイルを設定する）２番目の規則の方が優先されることになる。 また、３番目の作者­規則は !important にされている２番目の作者­規則の方が優先されることになる。 すなわち !important 宣言は，同じ作者スタイルシートの中でも機能する。 The first rule in the user’s style sheet in the following example contains an !important declaration, which overrides the corresponding declaration in the author’s style sheet. The declaration in the second rule will also win due to being marked !important. However, the third declaration in the user’s style sheet is not !important and will therefore lose to the second rule in the author’s style sheet (which happens to set style on a shorthand property). Also, the third author rule will lose to the second author rule since the second declaration is !important. This shows that !important declarations have a function also within author style sheets. 

/* 利用者のスタイルシート */
p { text-indent: 1em !important }
p { font-style: italic !important }
p { font-size: 18pt }

/* 作者のスタイルシート */
p { text-indent: 1.5em !important }
p { font: normal 12pt sans-serif !important }
p { font-size: 24pt }

6.3. CSS によるものでない呈示ヒントの優先順位

UA は、ソース文書のマークアップ内の呈示ヒント†を尊守してもよい。 例えば [HTML5] の中の bgcolor 属性や <s> 要素。 文書­言語に基づくすべてのスタイル付けは、対応する CSS 規則に転化された上で，［ UA レベルの規則, あるいは［ 詳細度ゼロを伴って作者スタイルシートの先頭に置かれた，作者レベルの規則 ］］として、カスケードの中に取り込まれなければならない。 文書­言語は、呈示ヒントが カスケードの中で UA ／ 作者 どちらのレベルに取り込まれるかを定義してもよい。 その場合、 UA は，それに従ってふるまわなければならない。 例えば [SVG11] では，その 呈示­属性 を作者レベルのものに対応付けている。 The UA may choose to honor presentational hints in a source document’s markup, for example the bgcolor attribute or <s> element in [HTML5]. All document language-based styling must be translated to corresponding CSS rules and either enter the cascade at the user agent level or be treated as author level rules with a specificity of zero placed at the start of the author style sheet. A document language may define whether a presentational hint enters at the UA or author level of the cascade; if so, the UA must behave accordingly. For example, [SVG11] maps its presentation attributes into the author level.

【† “presentational hints” — 本来は CSS が受け持つ所の，呈示のための役割も備えたマークアップ （参考：HTML が定義する呈示ヒント） 】

注記： カスケードの中で UA レベルに取り込まれる呈示ヒントは，作者スタイル, 利用者スタイルにより上書きできる。 カスケードの中で作者レベルに取り込まれる呈示ヒントは，作者スタイルにより上書きできるが、 !important でない利用者スタイルからは，上書きできない。 ホスト言語は、これらを考慮した上で，呈示ヒントに対する適切なレベルを選ぶべきである。 Note: Presentational hints entering the cascade at the UA level can be overridden by author or user styles. Presentational hints entering the cascade at the author level can be overridden by author styles, but not by non-!important user styles. Host languages should choose the appropriate level for presentational hints with these considerations in mind.

7. デフォルト法

カスケードの結果（カスケードの出力）が空である場合、他の何らかの方法で指定値が見出されなければならない。 継承プロパティは、その既定の値を，継承を通して親­要素から取り込む。 他のすべてのプロパティは、その初期値を既定の値としてとる。 スタイルシート作者は、 inherit や initial キーワードを通して，明示的に継承／初期化を要請できる。 When the cascade does not result in a value, the specified value must be found some other way. Inherited properties draw their defaults from their parent element through inheritance; all other properties take their initial value. Authors can explicitly request inheritance or initialization via the inherit and initial keywords.

7.1. 初期値

各種プロパティには、それぞれのプロパティ定義表にて，その 初期値 （ initial value ）が定義される。 与えられたプロパティ P が継承プロパティでなく, かつ 【与えられた要素­上に P の】 カスケード値が存在しない場合、 【その要素­上の】 P の指定値は， P の初期値で与えられる。 Each property has an initial value, defined in the property’s definition table. If the property is not an inherited property, and the cascade does not result in a value, then the specified value of the property is its initial value.

7.2. 継承法

プロパティ値は 継承 （ inheritance ）により，親­要素からその一連の子­要素へ伝播する。 要素­上のプロパティの 継承値 （ inherited value ）は、その要素の親­要素­上のプロパティの算出値で与えられる。 親­要素のないルート要素に対する継承値は、プロパティの初期値で与えられる。 Inheritance propagates property values from parent elements to their children. The inherited value of a property on an element is the computed value of the property on the element’s parent element. For the root element, which has no parent element, the inherited value is the initial value of the property.

（疑似要素における継承は、それぞれの疑似要素の定義に述べられる，仮想のタグ並び（ fictional tag sequence ）に従う。 [SELECT] ） (Pseudo-elements inherit according to a fictional tag sequence described for each pseudo-element [SELECT].)

一部のプロパティは、そのプロパティ定義表にて， 継承プロパティ （ inherited property ）であるものと定義される。 これは、カスケード値が存在しないときは，その値が継承を通して決定されることを意味する。 Some properties are inherited properties, as defined in their property definition table. This means that, unless the cascade results in a value, the value will be determined by inheritance.

プロパティを明示的に継承させることもできる。 inherit キーワードを見よ。 A property can also be explicitly inherited. See the inherit keyword.

注記： 継承は，文書­木に従うものであり、生成されるボックス木における組み替え — 合間に匿名ボックスが挟まれるなど — からは影響されないことに注意。 Note: Inheritance follows the document tree and is not intercepted by anonymous boxes, or otherwise affected by manipulations of the box tree.

7.3. 明示的なデフォルト法

各種 CSS 全域キーワードの意味は、以下に定義される。 これらの値をとるプロパティ宣言は、ある特定のデフォルト法のふるまいを明示的に指定する。 [CSS3VAL] にて指定される様に、すべての CSS プロパティは，これらの値を受容する。 Several CSS-wide property values are defined below; declaring a property to have these values explicitly specifies a particular defaulting behavior. As specified in CSS Values and Units Level 3 [CSS3VAL], all CSS properties can accept these values.

7.3.1 プロパティの再設定： initial キーワード

プロパティのカスケード値が initial キーワードになる場合、その指定値は，プロパティの初期値にされる。 If the cascaded value of a property is the initial keyword, the property’s specified value is its initial value.

7.3.2 明示的な継承： inherit キーワード

プロパティのカスケード値が inherit キーワードになる場合、そのプロパティの継承値が，指定値および算出値を与える。 If the cascaded value of a property is the inherit keyword, the property’s specified and computed values are the inherited value.

7.3.3 すべての宣言を取り消す ： unset キーワード

プロパティのカスケード値が unset キーワードになる場合、プロパティが継承プロパティであるならば inherit として扱われれ，そうでなければ initial として扱われる。 このキーワードにより、以前のカスケードに出現した すべての宣言値は取り消され、継承するかどうかもプロパティ（簡略形プロパティの場合は，そのすべての下位プロパティのそれぞれ）に応じて，適切に扱われる。 If the cascaded value of a property is the unset keyword, then if it is an inherited property, this is treated as inherit, and if it is not, this is treated as initial. This keyword effectively erases all declared values occurring earlier in the cascade, correctly inheriting or not as appropriate for the property (or all longhands of a shorthand).

【 “以前の…すべての” — カスケード法による順位付けの結果、出現順序による順位付けに従って， “以前の” 宣言値が取り消される（詳細度など，より高次の順位付けにおいて優先度が高い宣言値まで上書きされるわけではない）。 】

7.3.4 カスケードの巻戻し： revert キーワード

【 巻戻し（ rolling back ）： プロパティ値を，他の出自†による既定の値（他の出自にてそのプロパティに指定されている値）に “戻す” 。 （† 下の定義により、通常は，優先順位がより低い出自によるスタイル — important にされているもの対しては， “より高い” に逆転することになる。） 】

プロパティのカスケード値が revert キーワードになる場合のふるまいは、その宣言が属する出自に依存する： If the cascaded value of a property is the revert keyword, the behavior depends on the origin to which the declaration belongs:

UA 出自
unset に等価。 Equivalent to unset.
利用者 出自
カスケードを UA レベルに巻戻す： すなわち，プロパティの指定値は、プロパティに 作者レベル／ 利用者レベル の規則が指定されていないかのように計算される。 Rolls back the cascade to the user-agent level, so that the specified value is calculated as if no author-level or user-level rules were specified for this property.
作者 出自
カスケードを利用者レベルに巻戻す： すなわち，プロパティの指定値は、プロパティに作者レベルの規則が指定されていないかのように計算される。 revert の目的においては、この出自には 上書き／アニメーション 出自も含まれる。 Rolls back the cascade to the user level, so that the specified value is calculated as if no author-level rules were specified for this property. For the purpose of revert, this origin includes the Override and Animation origins.
